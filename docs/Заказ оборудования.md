## Что именно заказывать в Timeweb Cloud (вариант A, с двумя PostgreSQL)

### Таблица ресурсов

| Ресурс | Назначение / окружение | Что заказывать | Рекомендуемая конфигурация | Комментарий |
| :-- | :-- | :-- | :-- | :-- |
| **VPS \#1** | devtools + K8S‑master для dev/prod | VPS (Cloud‑серия) | **4 vCPU / 8 GB RAM / 80 GB NVMe** | Bitbucket Server, Pipelines Runner, Docker Registry, Maven/Docker/Helm/kubectl, Kubernetes control plane. |
| **VPS \#2** | K8S‑worker + dev‑PostgreSQL + pod’ы dev/prod | VPS (Cloud‑серия) | **2 vCPU / 4 GB RAM / 60 GB NVMe** | Worker‑нода Kubernetes, приложение greeting‑service (dev+prod), локальная PostgreSQL для dev. |
| **Managed PostgreSQL** | Основная БД prod | Managed PostgreSQL сервис | **1 vCPU / 2 GB RAM / 20 GB** | Prod‑БД с бэкапами и обновлениями на стороне Timeweb. |
| **VPC / сеть** | Общая сеть для всех ресурсов | VPC в Timeweb Cloud | /24 (например 10.10.0.0/24) | Создаётся через Terraform, отдельно не тарифицируется. |


***

### Пояснения по каждому ресурсу

**VPS \#1 — devtools + Kubernetes master (4 vCPU / 8 GB / 80 GB)**

- На нём живут все devtools‑компоненты из твоих схем:
    - **Bitbucket Server** (git‑репозитории);
    - **Bitbucket Pipelines Runner**;
    - **Docker Registry**;
    - CLI‑инструменты **Maven, Docker, Helm, kubectl**.
- Здесь же поднимаешь **control plane Kubernetes** (`kube-apiserver`, `controller-manager`, `scheduler`, `etcd`).
- 4 vCPU / 8 GB дают достаточный запас, чтобы одновременно крутить git, CI‑сборку (`./gradlew clean test`), docker‑build и мастер‑ноду, не утыкаясь в CPU/RAM.
- 80 GB диска достаточно для: репозиториев, кешей сборки, локального registry и логов, если периодически чистить старые образы и кеши.

**Итого по VPS \#1:** один сервер среднего размера, на котором сосредоточен весь devtools‑контур и управление кластером.

***

**VPS \#2 — worker + dev‑PostgreSQL (2 vCPU / 4 GB / 60 GB)**

- Это **единственная worker‑нода Kubernetes**:
    - на ней крутятся pod’ы **greeting‑service** для **dev** и **prod** (разные namespace, как в твоих схемах);
    - ресурсов хватит для 1 pod в dev и 2–3 pod в prod.
- Здесь же поднимаешь **локальную PostgreSQL для dev**:
    - отдельный экземпляр, на котором можно гонять миграции, ломать схемы, проверять тяжёлые запросы, не рискуя боевой БД.
- 2 vCPU / 4 GB достаточно для Spring Boot‑приложения, Kubernetes‑процессов ноды и dev‑PostgreSQL без серьёзной нагрузки; 60 GB — под образы, логи и данные dev‑БД.

**Итого по VPS \#2:** рабочая нода + dev‑БД, максимально приближённая к реальной prod‑схеме, но изолированная по данным.

***

**Managed PostgreSQL 1 vCPU / 2 GB / 20 GB — основная prod‑БД**

- Используется как **боевая БД для prod‑namespace**.
- Даёт:
    - автоматические бэкапы;
    - управляемые обновления;
    - отказоустойчивость на стороне провайдера.
- В приложение для prod пробрасываешь `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD` от managed‑сервиса.
- Логически это соответствует тому, что у тебя было на схемах как **`twc_database_cluster` + `twc_database_instance` + `twc_database_user`**.

**Итого по PostgreSQL:**

- **prod** ходит в **Managed PostgreSQL**;
- **dev** и эксперименты — в **PostgreSQL на VPS \#2**.

***

**VPC / сеть**

- Создаёшь одну **VPC /24** (например, `10.10.0.0/24`), внутри которой живут оба VPS и Managed PostgreSQL.
- Это соответствует блоку `twc_vpc greeting-service-vpc` на Terraform‑схеме.
- Оплачивается не как отдельный сервер, а как часть сетевой инфраструктуры (обычно без отдельной строки в счете).

***

### Короткий вывод

1. **Два VPS:**
    - VPS \#1: **4 vCPU / 8 GB / 80 GB** — devtools + K8S‑master.
    - VPS \#2: **2 vCPU / 4 GB / 60 GB** — K8S‑worker + dev‑PostgreSQL + pod’ы dev/prod.
2. **Две PostgreSQL:**
    - **Managed** (1 vCPU / 2 GB / 20 GB) — основная prod‑БД.
    - **Локальная на VPS \#2** — dev‑БД и полигон для экспериментов.
3. **Одна VPC** — общая сеть для всего этого хозяйства.

Если нужно, следующим шагом могу выдать конкретный список Terraform‑ресурсов (имена и параметры), точно соответствующий этой конфигурации.
